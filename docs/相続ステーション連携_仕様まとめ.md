# 相続ステーション連携 仕様まとめ

> **目的**：相続ステーション（既存社内システム、外部開発会社運用）と相続PF（本リポジトリ）の連携仕様の現状確定事項を1ファイルで共有する。
>
> 関連ドキュメント：
> - `docs/相続ステーション連携_項目マッピング表.xlsx`（先方共有用の項目対応表）
> - `docs/相続ステーション連携_API設計仕様書_v1.0.docx`（詳細API仕様。連携②を含む旧版、Phase 3の参考用）
> - `docs/相続ステーション連携_検討資料.docx`（方式選定の経緯）

---

## 1. 役割と全体像

| システム | 役割 |
|---|---|
| **相続ステーション** | LP担当が架電・面談アポ取り・案件管理・パートナー報告を行う既存システム |
| **相続PF（本リポジトリ）** | 面談アポ取得後の業務遂行（タスク管理・書類作成・請求・進捗）を担う新システム |

---

## 2. 連携スコープ（重要）

| 連携 | 方向 | 内容 | 今回スコープ |
|---|---|---|---|
| **連携①** | ステーション → 相続PF | 面談アポ取れた案件のデータを取り込み | **✅ 実装する** |
| **連携②** | 相続PF → ステーション | 受注／失注／完了＋確定売上の通知 | **❌ 今回スコープ外** |

→ **今回開発するのは連携①のみ**。連携②（PF→ステーション）は今回実装しない。

---

## 3. 発火タイミング

- ステーション側で案件ステータスが **「完了（アポ済）」** に変更された時
  - LP担当が依頼者と電話して面談アポを取得 → 手動でステータス変更
  - 「完了（アポ済）」＝ LP担当の架電業務における完了状態（アポ取得成功）
- 発火と同時に、相続PFのエンドポイントに HTTP 通知が飛ぶ

---

## 4. 通信仕様

### 4.1 通信方式
- **プロトコル**：HTTPS（TLS 1.2以上）
- **方向**：ステーション → 相続PF（Webhook 形式の push 型）
- **データ形式**：JSON、UTF-8
- **日付**：ISO 8601（例：`2026-05-14T10:30:00+09:00`）

### 4.2 HTTPメソッドの使い分け

| メソッド | 用途 | 動作 |
|---|---|---|
| `POST` | 初回新規送信 | PF側で新規案件作成 |
| `PUT` | 既存案件の更新 | PF側で**全項目上書き** |

- **PATCH（部分更新）は対応しない**（実装シンプル化のため）
- 重複送信時は **案件管理番号で Upsert**（既存があれば更新、なければ新規作成）

### 4.3 想定エンドポイント
- `POST /api/v1/integration/station/cases`（新規）
- `PUT  /api/v1/integration/station/cases/{caseNumber}`（更新）

## 5. 認証方式（先方と合意済）

| 要素 | 採用 | 備考 |
|---|---|---|
| **APIキー** | ✅ `Authorization: Bearer <key>` | 64文字ランダム、オーシャン側で発行 |
| **HMAC署名検証** | ✅ `X-Signature`（HMAC-SHA256） | ボディの改ざん検知 |
| **タイムスタンプ検証** | ✅ `X-Timestamp`（UNIX秒、±5分） | リプレイ攻撃対策 |
| **IP制限** | ❌ 初期は保留 | AWS設定変更とコスト増を避けるため |

### 5.1 鍵の本数

連携②がスコープ外なので、必要な鍵は **2本のみ**：

| 鍵 | 発行者 | 用途 |
|---|---|---|
| `INBOUND_API_KEY` | **オーシャン（PF）** | ステーション→PF 受信認証 |
| `INBOUND_HMAC_SECRET` | **オーシャン（PF）** | 同 署名検証 |

→ オーシャンで発行し、1Password 等で先方に渡す。

### 5.2 シークレット管理
- 環境変数または Supabase Vault で管理
- Git / メール / Slack 平文送付は厳禁
- 受け渡しは 1Password 等のシークレット共有ツール経由

## 6. 案件管理番号（※2026-06 改訂：2番号を併存）

> **改訂理由**：当初（v1.0）は「ステーション側の番号を *そのまま* `cases.case_number` に保存し、PF独自採番は使わない」方針だった。
> しかし相続PF側でも独自の採番ルールを運用することにしたため、**2つの番号を併存**させる方式に変更した。
> メインの案件管理番号は相続PFが採番したものを使う。

### 6.1 2つの番号
| 番号 | 列 | 採番元 | 用途 |
|---|---|---|---|
| **相続PF案件管理番号（メイン）** | `cases.case_number` | **相続PFが採番** | 画面・運用上のメイン番号 |
| **LP案件管理番号** | `cases.lp_case_number` | ステーション側 | 連携元の元番号。LP担当の検索・突合に使用 |

- **相続PFの採番ルール**：`YYMM` + 経路コード + 当日連番4桁（例 `2606LP0001`）。
  - `YY`=西暦下2桁 / `MM`=月2桁 / 経路コード=受注ルート(面談ルート)由来：LP経由=`LP` / 葬儀社経由=`SD` / HP経由=`HP` / 過去客経由=`PC` / 税理士経由=`ZE` / 当日連番=経路問わず当日作成順
  - 連携案件は受注ルートが `LP経由` になるため `26..LP....` 形式になる
- **ステーション側番号**（形式サンプル `SKN02101260113710`：SKN + 14桁数字）は新項目 `cases.lp_case_number`（LP案件管理番号）に保存。基本情報セクション（管理番号の次）と LP案件一覧で表示・検索（migration 078）。

### 6.2 突合キー（重要）
- **連携の突合キーはステーション側の番号**（Upsert時の突合・PUT時のURLパラメータ）。
- → 相続PF受信エンドポイントは **`lp_case_number` で突合 / Upsert** する（`case_number` ではない）。
- 新規取込時は、相続PFが上記ルールで `case_number` を別途自動採番して付与する。

### 6.3 先方（ステーション開発会社）への連携
- 先方の**送信処理は変更不要**（従来どおり自社の案件番号を「案件管理番号」として送ってもらう）。
- ただし共有済みのマッピング表は「案件管理番号 → `cases.case_number`」表記のため、**「→ `cases.lp_case_number`（突合キー）」へ訂正**して認識合わせする。
- 相続PF側の処理が変わるだけで先方実装に影響は出ないため**緊急度は低い**が、**結合テスト前／受信エンドポイント実装前**までに共有する。

### 6.4 確認待ち項目（次回打合せ）
- プレフィックス `SKN` の意味
- 14桁数字の内訳ルール
- 採番タイミング（架電登録時 / 完了（アポ済）時 のいずれか）
- 一意性の保証範囲（全期間 / 年度内）

## 7. 連携項目（マッピング）

詳細は `docs/相続ステーション連携_項目マッピング表.xlsx` 参照。サマリは以下：

| カテゴリ | 主な項目 |
|---|---|
| 案件識別 | 案件管理番号（#0、最重要キー） |
| 紹介情報 | 紹介元会社名 → PF側で partner_id 解決 |
| 被相続人情報 | 氏名／ふりがな／年齢／住所 |
| 連絡先情報 | 依頼者氏名／ふりがな／続柄／TEL×2／住所／備考 |
| 面談情報 | 伺い先住所／伺い先補足 |
| 相談情報 | ヒアリング内容／特記事項／その他ニーズ |

### 7.1 紹介情報・受注ルートの取り扱い（※2026-06 改訂）
- ステーション側の「会社名」1項目をそのまま受信
- **連携取込時の自動セット（確定ルール）**：
  | 項目 | 値 |
  |---|---|
  | `status`（ステータス） | `面談設定済` |
  | `order_route`（受注ルート＝面談ルート） | **`LP経由` 固定** |
  | `order_route_detail`（詳細） | 受信した紹介元会社名 |
  | `case_number` | 相続PFが自動採番（`2606LP....`） |
  | `lp_case_number` | ステーション側の元番号 |
- 受注ルートが `LP経由` になることで、LP案件一覧（受注ルート=LP経由で判定）に自動表示される
- 紹介元会社名は `order_route_detail` に格納。必要に応じ `referral_sources`（紹介元マスタ, migration 077）にも登録して候補を育てる
- 受注ルートの語彙は5経路（LP経由 / 葬儀社経由 / HP経由 / 過去客経由 / 税理士経由）に刷新済み。旧PFの「LP直/オーシャン直/その他＋詳細パートナー」2段階分類は **廃止**
- （旧記載：`cases.partner_id` 解決によるパートナーマスタ突合 → 現在は `order_route_detail`＋紹介元マスタ方式に変更）

### 7.2 データ受け入れ方針
- 弊社（PF）は **御社の入力値をそのまま受け取る**
- 形式・文字種・文字数による拒否は行わない
- 制約欄に書いた文字数等は **目安**
- 関係性／電話番号／ふりがな等もフリー入力で受け入れ

## 8. エッジケース対応（先方Q&Aで合意済）

| ケース | 対応 |
|---|---|
| 完了（アポ済）→ 別ステータスに変更 | **通知不要**。PF側は何もせず、必要なら運用担当が画面で手動修正 |
| 別ステータス → 再度完了（アポ済）に戻る | **Upsert** で吸収（案件管理番号で突合、既存あれば全項目上書き） |
| 項目変更（電話番号誤り訂正等） | **PUT で全項目上書き**（PATCH 非対応） |
| 既存案件（連携開始前の過去案件） | **連携対象外**、何もしない |

### 設計思想
**「シンプルにできるところは全部シンプルにする」** を貫いた結果：

- ✅ 作る：POST/PUT、Upsert、案件管理番号突合
- ❌ 作らない：キャンセル通知受信、PATCH差分更新、上書き保護管理表

工数最小化のためレアケースは手動運用でカバー。
