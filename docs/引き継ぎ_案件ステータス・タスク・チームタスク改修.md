> 実装サマリは末尾「実装結果」を参照。既存の実装状況（migration 050〜054 等）は「現状」を参照。

## 1. 案件ステータスの変更

- **新設：「検討中（契約書待ち）」**
  - 面談その場で契約できず、後日お客様が契約書を送付してくる待ち状態。
  - 並び順は「検討中」の直後。
  - 期限は回答予定日でリマインド（目安 1週間）。
- **改称（表示ラベルのみ・内部キーは維持して既存ロジックを壊さない）**
  - 受注 → **受託**
  - 失注 → **不受注**
  - 架電案件化 → **新規**
- 「検討中」は**単一のまま**（細分化しない）。検討期間は **お客様回答予定日** で管理。
- **お客様回答予定日を必須化**（未設定ケースを無くす）。

実装メモ：`CASE_STATUSES`（src/lib/constants.ts）と `CaseStatus` 型に「検討中（契約書待ち）」を追加。改称は **表示用のラベルマップ**を用意し、`status === '受注'` 等のキー比較ロジックは触らない。status_definitions のマイグレーションも追加。

## 2. システムタスク（最終定義）

| タスク | 担当 | 起点 → 標準期限 | 生成トリガー・備考 |
|---|---|---|---|
| 管理担当をアサインする | 管理担当 | 受注日 +1日 | 受注3日経過＆管理担当なしでアラート（既存 sys_assign_manager） |
| 案件内容の共有 | 受注担当 | 管理担当アサイン日 +3日 | **生成＝案件に管理担当がアサインされた時**（受注時ではない） |
| 検討状況の確認 | 受注担当 | お客様回答予定日 | ステータス→検討中で生成 |
| 面談メモを記載する | 受注担当 | 面談予定日 **当日** | 面談予定日超過＆面談実施日未記録 |
| **案件詳細入力（オーダーシート）** | 受注担当 | 受注日 **+3日** | **新規追加**。面談メモの後ろ。受注時生成 |
| 前受金請求書作成・送付 | 両担当 | 受注日 当日 | 受注時生成 |
| 前受金入金の確認 | 両担当 | **請求書タブの入金予定日**（祝日補正なし＝単純な日付） | 受注時生成 |
| お客様への御礼連絡 | 両担当 | **前受金入金の確認 完了日** | **上流タスク完了時に生成/期限確定**（タスク依存） |
| 原本預かり証の送付 | 両担当 | **期限なし** | 受注時生成 |
| 初期タスクあげ | 両担当 | 管理担当アサイン日 +1日 | |
| 定期進捗連絡 | 両担当 | 前回連絡 +7日 | 日次cron（既存 sys_periodic_contact） |
| 今週の進捗報告 | 管理担当 | 前回確認済 +7日 | 日次cron（既存 sys_weekly_report） |
| ~~送付書類の確認~~ | — | — | **削除** |
| ~~契約書の作成・送付~~ | — | — | **削除** |
| ~~契約書の確認~~ | — | — | **削除** |

### 担当・完了の扱い
- **両担当** ＝ 1タスクに **受注担当・管理担当の2人を自動アサイン**（生成時に案件の各担当者を `task_assignees` に紐付け）。
- マイページのタスクは **担当者ベース表示**（自分が担当者の未完了タスク）。受注担当ページ＝受注担当タスク、管理担当ページ＝管理担当タスク（＝自分が担当のもの）。
- 「どちらかが完了したら完了」は **1タスク共有なので自動的に成立**。
- **引き取り**：着手者(`started_by`)＝引き取った人 ＋ ステータス「対応中」。表示は **「対応中（名前）」** のまま残し、**完了で消える**（個人・両担当とも同方式）。

### システムタスク生成時の自動アサインのルール
- 受注担当固定タスク → 案件の受注担当へ。
- 管理担当固定タスク → 案件の管理担当へ。
- 両担当タスク → 受注担当＋管理担当の両方へ。
- 該当ロールが未アサインの場合の穴（誰にも紐付かない）への対応は要検討（フォールバックで案件ロールで拾う等）。

## 3. チームタスク（チームダッシュボードに新設）

- **1つの「チームタスク」欄に統合**（役割で分けない）。チーム案件の **未完了システムタスクのうち、期限超過＋あと1〜2日で期限** のものを表示。順調なものは出さない。
- 各タスクに **担当区分ラベル（受注担当 / 管理担当 / 両担当）** を表示。
- **引き取りは (B)ソフト制御**：全員ボタンは押せる。ラベルで「誰が拾うべきか」を誘導するだけ（システムでのブロックはしない）。
- 表示場所：受注担当・管理担当 **両方のチームダッシュボードに同じ統合チームタスク欄**を出す（チーム全員が気づける）。
- 引き取り＝着手者＝自分＋対応中、「対応中（名前）」表示、完了で消える。

## 4. タスク作成UI（新規）

- **マイページに「タスク作成」ボタンを追加**し、作成フォームを開く。
- フォーム項目：
  - タスク名
  - フェーズ（任意・テンプレートから選択。システムタスク用に **Phaseなしも可**）
  - **担当者（個人）**
  - **担当チーム**（チーム割り当て＝チームタスクに出る基盤）
  - 期限（任意・未入力なら期限なし）
  - 優先度
- 案件詳細のタスクタブからの作成は既存。マイページからも作れるようにする。

### 残課題・要確認（staging反映後）
- 各DBトリガー（受注時生成／管理担当アサイン時生成／御礼連絡依存生成／invoices期限追従）の実挙動はブラウザ未確認。型/ビルドのみ検証済み。
- フェーズ3で `task_assignees` ベースに切替済のため、システムタスクが担当者に正しく自動アサインされていることが前提（migration 056 の backfill で既存分も補完）。
- 「管理担当をアサインする」タスクは担当不在のため実装上 **受注担当(sales)** にアサイン（仕様表の「管理担当」は担当不在で実行不能のため）。
- チームタスク欄の「チーム案件」は受注担当メンバーの担当案件ベース（teamCaseIds）＋ `tasks.team_id` 指定分。
