# 受注区分 → 業務 → 作業 マスタ化 仕様（検討まとめ）

最終更新: 2026-06-16 / ステータス: **合意形成中（実装前）**

## 0. ねらい（なぜ直すか）
- 今は「受注見込み手続き区分（複数選択）」と「役割分担の業務（フリー選択）」が**バラバラ**で、受注区分と業務が紐づいていない。
- オーダーシートも対応中のタブも、**受注区分に関係なく全部出る**。
- 業務名に**表記ゆれ**があり、既存の事務管理タスクテンプレと業務の対応も曖昧。

→ **「受注区分を1つ選ぶと、やる業務が決まり、業務ごとの作業（誰が・どの書類を・いつ請求/用意するか）を管理できる」** 形にする。

## 1. 基本モデル（3層 ＋ タスク/タブの役割分担）

```
① 受注区分（案件に1つ）        … 何の業務が要るかを決める
   └ ② 業務（区分のプリセット・外せる） … タブ／OSセクションの出し分け単位
        └ ③ 作業（業務のプリセット）   … 誰が・どの書類を・いつ請求/用意するか ＝実体管理の核
```

- **タブ＝実体・進捗**（どの戸籍を・どこに請求・届いたか 等）。
- **タスク＝「誰が・いつまでに」やるか**（割り当て・締切）。実体はタブ側が持つ。
- タスクと取得物の関係は **1対1でない**：
  - A. 取得物が届いて→その場でタスク作成・着手（受信簿の「タスクに着手」）
  - B. 先にタスクを用意→届いたら紐づけて開始
  - C. 取得物と無関係に進める（相関図作成・協議書作成 等）

## 2. 受注区分（10）
`手続き一式 / 登記 / 遺言 / 信託 / 放棄 / 調停 / 検認 / 後見 / 契約書 / 執行`
- **単一選択**（複数不可）。
- **「手続き一式」は独立した区分**（登記〜執行の全部、ではない）。

## 3. 業務マスタ（表記統一済み・全20）
**表記ゆれ統一**: `不動産/査定`→不動産（査定は作業）／`遺言作成(自筆)(公正証書)`→遺言作成／`信託契約書作成(私文書)(公正証書)`→信託契約書作成。

調査・共通系: 戸籍 / 相関図 / 法定相続情報取得 / 不動産 / 金融資産 / 目録
手続き系: 協議書 / 登記 / 解約 / 手紙 / 遺言作成 / 信託契約書作成 / 放棄手続き / 調停手続き / 検認手続き / 後見手続き / 契約書作成 / 執行通知
独立・区分非依存: **相続税**（③A）/ **経理**（精算・請求・入金・納品・クローズ）

※ ③A 決定: `年金照会`・`負債調査` は **金融資産の作業**に畳む。`相続税` は **独立業務**として追加。

## 7. これから行う修正（フェーズ計画）
- **Phase 1 マスタ整備**: 受注区分(10)・業務(20・正規化)・作業(エクセルD列)の定義をコード/DBに。受注区分→業務、業務→作業のプリセット。task_templates に業務キー付与。
- **Phase 2 受注内容UI**: 「手続区分(複数)」→「受注区分(1つ)」。役割分担を「区分プリセット業務(全選択→外す)→作業ごとに担当(既定=自社)」に。
- **Phase 3 タブ/OS出し分け**: 選択業務でタブ・OSセクションを出し分け。新規業務(信託/放棄/調停/検認/後見/契約書/執行通知/手紙)の汎用「手続き作業表」UI。
- **Phase 4 作業↔書類/請求/受信簿の統合**: 各作業に必要書類・請求・受領(受信簿連動)を紐づけ。既存の取得区分/受信簿と統合。
- **Phase 5 タスク連携**: 自社作業からタスク生成、A/B/C 3パターン対応。

## 8. データモデル方針（案）
- `case.order_category`（受注区分・単一）。
- `case_gyomu`（案件×選択業務）。`case_sagyo`（案件×作業：業務/担当/書類/請求 …）。
- もしくは intake_roles を発展させて 業務/作業/担当 を構造化。
- 既存の koseki_requests / financial_assets / real_estate_properties / contract_documents は「作業の実行インスタンス（取得物）」として温存。
- マイグレは追加のみ・段階適用（ユーザーが手動適用）。
