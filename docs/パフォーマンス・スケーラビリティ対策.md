# パフォーマンス・スケーラビリティ対策

最終更新: 2026-04-09

## 概要

本ドキュメントは、相続プラットフォームにおける **同時接続時のパフォーマンス確保** と **スケーラビリティ対策** をまとめたものである。
開発実装が一定進んだ段階（βリリース前後）で、本ドキュメントの各項目をチェックリストとして反映・検証する。

### 用語について

この種の対策の一般的な呼称は以下の通り:

| 呼称 | ニュアンス |
|---|---|
| **パフォーマンスチューニング** | 最も一般的。処理速度・応答時間の最適化全般 |
| **スケーラビリティ対策** | 同時利用者数の増加に耐える設計 |
| **負荷対策 / 負荷軽減処理** | 口語的表現。意味は同じ |
| **性能最適化** | ややフォーマルな表現 |

本プロジェクトでは **「パフォーマンス・スケーラビリティ対策」** を正式名称とする。

---

## 想定同時利用規模

- 想定ユーザー数: **50〜100人の同時利用**
- 主な操作: 案件一覧閲覧、案件詳細編集、書類アップロード、AI抽出、請求書作成
- ワークロード特性: **読み込み > 書き込み**（相続案件の性質上、参照系が多い）

実効的な同時クエリ数はピークで **10〜20 req/sec** 程度を見込む。

## 1. Supabase 接続設定

### 1.1 プラン選定

| プラン | Direct接続 | Pooler接続 (Transaction) | 判断 |
|---|---|---|---|
| Free | 60 | 200 | ❌ 本番不可 |
| **Pro ($25/月)** | 90 | **400** | ✅ **採用** |
| Team | 120 | 600 | 将来検討 |

**決定: Proプランを本番環境で使用する。**

### 1.2 Connection Pooler の使用（必須）

Next.js (サーバレス) から接続する場合、**必ず PgBouncer (Connection Pooler) を経由**する。Direct接続はサーバレス関数起動ごとに接続を食い潰し、ピーク時に接続数上限に達する。

- ❌ Direct: `db.xxx.supabase.co:5432`
- ✅ **Pooler (Transaction mode)**: `aws-0-xxx.pooler.supabase.com:6543`

環境変数 `DATABASE_URL` は Pooler の URL を設定する。

## 2. クエリ設計

### 2.1 N+1 クエリの回避

案件一覧画面などで、案件ごとに相続人・タスク・書類を別クエリで取得する N+1 を避ける。Supabase JS クライアントの埋め込み SELECT を活用する。

```ts
// ❌ BAD: N+1
const cases = await supabase.from('cases').select('*');
for (const c of cases.data) {
  const heirs = await supabase.from('heirs').select('*').eq('case_id', c.id);
}

// ✅ GOOD: 1クエリ
const cases = await supabase
  .from('cases')
  .select('*, heirs(*), tasks(*), documents(*)');
```

### 2.2 インデックス設計

以下の列には必ずインデックスを作成する:

- `cases.assigned_user_id`, `cases.status`, `cases.created_at`
- `tasks.case_id`, `tasks.assigned_user_id`, `tasks.due_date`
- `heirs.case_id`
- `documents.case_id`, `documents.type`
- `invoices.case_id`, `invoices.status`, `invoices.issued_at`
- `partner_payouts.partner_id`, `partner_payouts.month`

**RLS ポリシーで参照する列には必ずインデックスを張る**（重要）。

### 2.3 スロークエリの特定

- Supabase Studio の **Database > Query Performance** で遅いクエリを可視化
- `pg_stat_statements` 拡張を有効化しておく
- 本番投入前に主要画面で `EXPLAIN ANALYZE` を実行し、Seq Scan が発生していないか確認

## 3. RLS (Row Level Security) 最適化

### 3.1 原則

- 全テーブルで RLS を有効化する（セキュリティ要件）
- **RLS の `USING` 句は、インデックスが効くシンプルな条件にする**
- `auth.uid()` の多段 JOIN は避ける。必要ならビューまたはヘルパー関数を作り `SECURITY DEFINER` で回避する

### 3.2 アンチパターン

```sql
-- ❌ BAD: 複雑なサブクエリで毎回JOINが走る
CREATE POLICY "case_access" ON cases
USING (
  id IN (
    SELECT case_id FROM case_members WHERE user_id = auth.uid()
    UNION
    SELECT case_id FROM partner_cases WHERE partner_id IN (
      SELECT id FROM partners WHERE owner_user_id = auth.uid()
    )
  )
);
```

### 3.3 推奨パターン

```sql
-- ✅ GOOD: 事前計算したフラットな権限テーブルを参照
CREATE POLICY "case_access" ON cases
USING (
  id IN (SELECT case_id FROM v_user_case_access WHERE user_id = auth.uid())
);
-- v_user_case_access はマテリアライズドビューまたは通常ビューで事前に結合
```

## 4. Realtime 購読の制御

Supabase Realtime は WebSocket 接続を維持するためリソースを消費する。

- **購読は必要な画面・必要なテーブルのみに限定**
- 案件詳細画面を開いているときだけ `cases`, `tasks`, `documents` を購読する
- 画面離脱時に `unsubscribe()` を確実に呼ぶ（メモリリーク防止）
- ダッシュボードなど一覧系は Realtime ではなく **ポーリング（30秒〜1分間隔）** でよい

---

## 5. Storage 最適化

戸籍謄本・遺産分割協議書などの PDF は容量が大きくなりやすい。

- **Supabase Storage + CDN キャッシュ** を活用
- 書類一覧はサムネイル URL を DB に保存し、本体 PDF は必要時にのみ取得
- 一括ダウンロードは避け、必要なファイルのみストリーミング取得
- アップロードは **クライアント直アップロード (signed URL)** を使い、Next.js API サーバを経由させない

---

## 6. Next.js / フロントエンド側

### 6.1 データ取得戦略

- **Server Components (RSC) でのデータ取得を優先** — クライアントでの不要なウォーターフォールを防ぐ
- ISR / `revalidate` を活用し、マスタ系（金融機関一覧、パートナー一覧など）はキャッシュ
- 一覧画面は **ページネーション or 無限スクロール** で、1回の取得を最大 50 件程度に制限

### 6.2 画像・アセット

- `next/image` で自動最適化
- 静的アセットは Render の CDN キャッシュを活用

---

## 7. 外部 API 連携のボトルネック対策

Supabase 以外に負荷が発生しうる箇所:

### 7.1 Claude API

- **Anthropic API のレート制限** に注意（Tier によって RPM / TPM が変動）
- 戸籍 AI 抽出などの長時間処理は **非同期ジョブ化** し、キュー（Supabase Edge Functions + pgmq もしくは Render ワーカー）経由で実行
- 結果は DB に保存し、フロントはポーリング or Realtime で受信

### 7.2 Stripe Webhook

- **冪等性処理（idempotency key 管理）** を必ず実装
- Webhook 受信用エンドポイントは軽量に保ち、重い処理は非同期ジョブへ流す
- 署名検証は必須
