### 2-2. Supabase Storage セットアップ（ユーザーに実行してもらう SQL）

```sql
-- avatars バケット作成
INSERT INTO storage.buckets (id, name, public)
VALUES ('avatars', 'avatars', true)
ON CONFLICT (id) DO NOTHING;

-- 認証ユーザーは誰でもアップロード可（セキュリティはユルめ方針）
DROP POLICY IF EXISTS "avatars_authenticated_upload" ON storage.objects;
CREATE POLICY "avatars_authenticated_upload" ON storage.objects
  FOR INSERT TO authenticated
  WITH CHECK (bucket_id = 'avatars');

DROP POLICY IF EXISTS "avatars_authenticated_update" ON storage.objects;
CREATE POLICY "avatars_authenticated_update" ON storage.objects
  FOR UPDATE TO authenticated
  USING (bucket_id = 'avatars');

DROP POLICY IF EXISTS "avatars_authenticated_delete" ON storage.objects;
CREATE POLICY "avatars_authenticated_delete" ON storage.objects
  FOR DELETE TO authenticated
  USING (bucket_id = 'avatars');

-- 公開読み取り
DROP POLICY IF EXISTS "avatars_public_read" ON storage.objects;
CREATE POLICY "avatars_public_read" ON storage.objects
  FOR SELECT USING (bucket_id = 'avatars');
```

## 3. 将来の Phase B：サンクスポイント機能

### 3-1. 構想

「誰かに助けてもらったら、その人にサンクスポイントを送れる仕組み。月間で多い人に社長が何かをプレゼント」

### 3-2. データモデル（migration 030 で追加予定）

```sql
CREATE TABLE thanks_points (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  from_member_id UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  to_member_id   UUID NOT NULL REFERENCES members(id) ON DELETE CASCADE,
  message        TEXT NOT NULL,
  case_id        UUID REFERENCES cases(id) ON DELETE SET NULL,  -- 関連案件（任意）
  created_at     TIMESTAMPTZ DEFAULT now(),
  CHECK (from_member_id <> to_member_id)
);
CREATE INDEX idx_thanks_to ON thanks_points(to_member_id, created_at DESC);
CREATE INDEX idx_thanks_from ON thanks_points(from_member_id, created_at DESC);
```

### 3-3. 機能要素（Phase B 着手時に詳細化）

1. **送信モーダル**：他人のプロフィールに「サンクスを送る」ボタン → メッセージ入力 → 送信
2. **プロフィール表示**：
   - 「もらったサンクス（今月）」カウンター
   - 最近受け取ったメッセージ一覧
   - 「送ったサンクス」も表示すべきか要相談
3. **TOPページに「今月のサンクスランキング」カード**：トップ3〜5人
4. **管理者向けの月次集計画面**：社長がMVP決めるとき用

### 3-4. Phase A でやるプレースホルダー

プロフィール画面に「もらったサンクス」枠だけ用意して、`今月: -- 件` 表示 + 「Phase B で実装予定」と注記。Phase B 着手時にここを実装。

## 5. 直前のセッションで決めたデザイン方針（再確認）

- **配色**：白ベース + Ocean ネイビー（#0f487e）アクセント。`brand-*` の Tailwind ユーティリティで適用。
- **絵文字は使わない**。Lucide の SVG アイコン。
- **フォント**：Inter（英数）+ Noto Sans JP（日本語）+ JetBrains Mono（等幅）。
- **アイコン**：色付き丸 + 頭文字（実画像があるならそれを優先）。
- **共通コンポーネント**：`Button` / `Card` / `Badge` / `<UserAvatar>`（Phase A で追加）
- **「上品さ」を意識**：派手な装飾は避ける。グラデーション・shadow・ring は控えめに。

## 6. 注意点・既知の癖

- **Tailwind 4 の `@theme inline`** で brand 色を定義済み。CSS 変更後は `.next/` キャッシュクリアが必要なケースがあった。
- **Windows の git CRLF 警告**は無視してOK（`core.autocrlf=true`）。
- **localhost dev 起動時**：worktree の `.env.local` がない場合、メインフォルダからコピーが必要。
- **ポート3000が使われがち**：`autoPort: true` で自動で別ポートに割り振られる。
